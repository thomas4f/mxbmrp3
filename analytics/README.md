# Analytics

A static dashboard derived from the plugin's [Aptabase](https://aptabase.com) telemetry.
[`REPORT.md`](REPORT.md) (plus the SVG charts in [`charts/`](charts/)) is generated by
[`tools/analytics_report.py`](../tools/analytics_report.py) from the **monthly Parquet
exports** Aptabase provides. The raw exports are **not** kept in the repo - only the
generated report is.

## Regenerate

1. In Aptabase, export the month(s) you want as Parquet.
2. Run the generator (Windows `&`, or one command per line):

   ```
   python3 tools/analytics_report.py path\to\mxbmrp3release2026XX.parquet ...
   ```

   Pass any number of files (or a glob). They're merged and de-duplicated, and the report's
   data window is taken from the data itself, not the filenames. Output defaults to
   `analytics/`; override with `--out`.

3. Commit the regenerated `analytics/REPORT.md` + `analytics/charts/*.svg`. Do **not** commit
   the `.parquet` (it's git-ignored).

Requires `pandas` + `pyarrow` (`python3 -m pip install -r tools/requirements.txt`). The SVG
charts have no extra dependency (`tools/analytics_svg.py`). Run `--selftest` to exercise the
whole pipeline on synthetic data with no export on hand.

## What it measures (and how to read it)

It opens with a short **Highlights** TL;DR (reach, platform, repeat use, top HUD, stability),
then the detail: what matters to the **plugin dev / users** (installs, activity per game,
versions, geography, OS, repeat usage, feature/HUD/widget adoption) and to the **upstream game
dev** (crash rate, crash categories, the most common crashes with triggers + workarounds, and
an uncatalogued tail). Each crash is listed once - named crashes by name, the rest by fault
signature - so the two crash tables don't overlap. Count charts label each bar with its count
and share; adoption charts label with share then count. Two things are worth internalising:

- **`install_id`, not `user_id`.** Aptabase's `user_id` is a privacy-preserving hash that
  **rotates daily** (~8 distinct ids per real install in the sample data), so it badly
  over-counts installs. Every install-level metric keys on the plugin's own stable
  `install_id` from `string_props`.

- **Coverage-aware by design.** Telemetry landed in **1.26** and was refined through **1.27**,
  so early builds omit fields (e.g. `os_version`). Rather than average over a denominator that
  doesn't include those events, each feature/geo/OS metric reports the value **and** how many
  installs actually sent the field. The *About this data* section spells out the gaps per
  release line, so a partial rollout is never mistaken for a real trend.

- **Adoption is per-game.** HUDs/widgets/features are game-specific (ECU & Tyre Temp are GP
  Bikes only; FMX/Records are MX Bikes only), and the plugin only emits a flag where that
  item exists. So the adoption charts are computed for the **primary (most-installed) game**
  - otherwise a GP-Bikes-only widget's rate would sit next to MX-Bikes rates over a totally
  different base. Within that one game, a flag reported by fewer installs (a newer feature) is
  shown as *"N of M"* so its smaller base is explicit.

## Crash linkage

Crash events carry a faulting `module+offset` and the game build hash. The generator joins
these against [`crash_analysis/known_game_crashes.json`](../crash_analysis/known_game_crashes.json)
(the machine-readable crash registry) on `(game_build, offset)` and annotates matched crashes
with their catalogued name + whether a workaround exists. When a new build is catalogued there,
re-running this tool automatically classifies more of the crash tail.

Crash stats are computed over plugin **1.27.5+ only** - the release that made crash telemetry
fully instrumented: it added the faulting-thread backtrace and the access-violation type on top
of the plugin/game version pinned in 1.27.0. Before 1.27.5 crashes were under-reported and carry
no stack/av_type, so restricting to 1.27.5+ keeps the whole crash population consistent (rate,
categories, av_type and stacks all from one set). The rate is expressed **per launch, not per
session**: crashes are always sent but `session_end` is sampled, so launches are the only honest
denominator. Crashes whose host process is dev/replay tooling are excluded too. The threshold is
a single constant (`CRASH_MIN`) - bump it as newer instrumentation lands.
